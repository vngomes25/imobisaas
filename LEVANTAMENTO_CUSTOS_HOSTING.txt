━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  │ MySQL           │ 8.0                                      │

  3. OPÇÃO A — MANUS (Plataforma original)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O Manus é uma plataforma de agente de IA que pode construir e publicar
  aplicações web temporariamente durante o desenvolvimento.

  IMPORTANTE: O Manus NÃO é uma plataforma de hospedagem de produção.
  Os ambientes criados no Manus são temporários (uso durante o desenvolvimento).
  Para deixar o sistema disponível 24/7 para um cliente real, é necessário
  migrar para uma hospedagem dedicada.

  Planos Manus (referência — verificar preços atuais em manus.im):
  ┌────────────────┬──────────────────────────────────────────────────────┐
  │ Gratuito       │ Créditos limitados, sem SLA, ambiente temporário     │
  │ Pro (~US$39/m) │ Mais créditos, execução mais longa, sem persistência │
  │                │ de banco de dados contínua para produção              │
  └────────────────┴──────────────────────────────────────────────────────┘

  VEREDICTO: ❌ Não recomendado para produção.
  Use o Manus apenas para desenvolvimento e prototipagem.
  Custo de produção em outras plataformas é menor e mais estável.

  4. OPÇÃO B — RAILWAY.APP  ⭐ RECOMENDADO (melhor custo-benefício inicial)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Site: railway.app  |  Cobrança: por uso (GB RAM × horas)

  O Railway é uma PaaS (Platform as a Service) moderna. Você sobe o código
  via GitHub e ele cuida do resto: deploy automático, MySQL gerenciado,
  domínio com SSL, logs, variáveis de ambiente.

  O que é incluído:
  • Serviço Node.js (backend + frontend servido pelo Express)
  • MySQL 8 gerenciado (backups automáticos)
  • Domínio .railway.app gratuito ou domínio próprio com SSL automático
  • Deploy automático via GitHub Push
  • Logs em tempo real, métricas básicas

  Estimativa de custo para o ImobiSaaS:
  ┌──────────────────────────────┬───────────────────────────────────────┐
  │ Node.js service (512 MB RAM) │ ~US$ 5 – 8 / mês                     │
  │ MySQL 8 gerenciado (1 GB)    │ ~US$ 5 – 7 / mês                     │
  │ Domínio próprio + SSL        │ US$ 0 (SSL) + custo do domínio        │
  │ ──────────────────────────── │ ───────────────────────────────────── │
  │ TOTAL ESTIMADO               │ US$ 10 – 15 / mês ≈ R$ 55 – 85/mês  │
  └──────────────────────────────┴───────────────────────────────────────┘

  Prós:
  ✅ Sem gerenciar servidor (sem SSH, sem updates manuais)
  ✅ MySQL já configurado e com backup
  ✅ Deploy em minutos via GitHub
  ✅ Escala fácil se crescer
  ✅ Interface moderna e fácil de usar
  ✅ Suporte a múltiplos projetos (multi-cliente fácil)

  Contras:
  ❌ Preço em dólar (varia com câmbio)
  ❌ Cobrança por uso (pode surpreender se mal configurado)
  ❌ Plano gratuito descontinuado (trial de US$5 apenas)

  14. CENÁRIO 2 — SAAS: VENDER PARA MÚLTIPLOS CLIENTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Existem duas abordagens para atender múltiplos clientes:

  ──────────────────────────────────────────────────────────────────────────
  ABORDAGEM A — Uma instância por cliente (isolamento total)
  ──────────────────────────────────────────────────────────────────────────

  Cada cliente tem seu próprio servidor, banco e domínio.

  Exemplo com Hostinger KVM 1 por cliente:
  ┌────────────────┬────────────────────────────────────────────────────────┐
  │ 1 cliente      │ R$ 43/mês (infraestrutura)                            │
  │ 3 clientes     │ R$ 129/mês (3 VPS separados)                         │
  │ 5 clientes     │ R$ 215/mês                                            │
  │ 10 clientes    │ R$ 430/mês                                            │
  └────────────────┴────────────────────────────────────────────────────────┘

  Prós: Isolamento total, um cliente não afeta o outro, fácil de fazer backup.
  Contras: Custo cresce linearmente, mais trabalho de manutenção.

  MELHOR PARA: Até 5 clientes. Simples de gerenciar.

  ──────────────────────────────────────────────────────────────────────────
  ABORDAGEM B — Multi-tenant: todos no mesmo servidor (eficiente)
  ──────────────────────────────────────────────────────────────────────────

  Um servidor mais robusto atende a todos. Cada cliente tem seu próprio
  banco de dados (schemas separados) dentro do mesmo MySQL.

  Requer adaptação no código:
  • Campo "tenantId" em todas as tabelas
  • Autenticação separada por domínio/subdomínio
  • Painel de controle central para o administrador do SaaS

  Exemplo: Hetzner CX32 (8 GB RAM, 4 vCPU) para até 15 clientes:
  ┌────────────────────────────────────────────────────────────────────────┐
  │ Hetzner CX32 (até 15 clientes)   │ ~R$ 47/mês total                  │
  │ Custo por cliente (15 clientes)  │ R$ 3,13/mês                       │
  └────────────────────────────────────────────────────────────────────────┘

  MELHOR PARA: 6+ clientes. Precisa de mais desenvolvimento inicial.

  ──────────────────────────────────────────────────────────────────────────
  ABORDAGEM C — Híbrida (RECOMENDADA para crescimento)
  ──────────────────────────────────────────────────────────────────────────

  • Fase 1 (0–5 clientes): Um VPS por cliente (simples, rápido de entregar)
  • Fase 2 (5–20 clientes): Migrar para servidor maior multi-tenant
  • Fase 3 (20+ clientes): AWS/GCP com auto-scaling
